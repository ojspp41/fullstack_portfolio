# AI 사용량·비용 미터링 & 과금 백오피스

> **FE 직접 구현** — 대시보드 · 목록 조회 공통 훅 · ExcelJS 멀티시트 내보내기 · 필터 6종 URL 동기화
> **BE 직접 구현** — 미터링 조회 API(필터 6종·페이지네이션·엑셀) · 사용 한도 제어 API(요금제별 쿼터, 실시간 카운터 차단)
> **설계 이해·연동** — Go 파이프라인(Kafka 인입 · 일배치 사전집계 · 멱등 · SCD-2 가격/환율 이력)

## S — 상황

AI 서비스가 커지면서 '누가 얼마 쓰고 얼마 청구되는지'를 관리자가 보고 통제할 백오피스가 필요했다. 기능 세 개를 따로 붙이는 문제가 아니라 **하나의 파이프라인으로 이어져야 하는 문제**였다.

- 호출 기록은 Kafka로 들어오는데, 최소 한 번 전달(at-least-once) 특성상 같은 기록이 두 번 이상 들어올 수 있다.
- 매번 수백만 건 원본을 즉석 계산하면 느리고 비싸다.
- 가격·환율은 시간에 따라 바뀌는데, 예전 값을 지우면 과거 청구를 재현할 수 없다 (정산·감사 불가).
- 한도 없이 호출되면 비용이 통제 불능이 된다.

## T — 과제

1. 중복이 섞여 쏟아지는 호출 기록을 **두 번 처리해도 안전하게(멱등)** 집계
2. 대시보드가 추가 가공 없이 바로 그릴 수 있는 **요약 데이터로 사전집계**
3. 가격·환율이 바뀌어도 **과거 청구를 그 시점 기준으로 재계산·검증**
4. 사용량이 요금제 한도를 넘으면 **호출 길목에서 실시간 차단·경고**

## A — 행동

### 내보내기 End-to-End 흐름 (요청부터 파일까지 직접 구현)

```
[FE] 내보내기 클릭 → [BE] 미터링 조회 API (필터 6종·페이지 처리)
  → [BE] ExcelJS 스트리밍 (요약 + 상세 시트) → [FE] 파일 다운로드
```

**BE**
- 브라우저에서 통째로 만들던 대용량 엑셀을 **서버 스트리밍으로 전환** — ExcelJS 스트리밍 라이터로 행을 받는 즉시 흘려보내고, 데이터 소스도 페이지 단위 조회로 메모리를 행 수와 무관하게 평탄화
- 필터 6종을 하나의 조회 쿼리로 조합, 요약 + 상세 멀티시트 구성
- 사용 한도 제어: 요금제 한도 초과 시 게이트웨이의 **Redis 카운터가 호출이 나가기 전 429로 즉시 차단·경고**

**FE**
- 필터 6종을 URL에 저장 — 새로고침·링크 공유해도 같은 화면 유지
- 목록 조회 공통 훅으로 로딩·에러·페이지네이션 재사용
- 내보내기 진행/완료/실패 상태를 화면에서 처리

**파이프라인 (설계 이해·연동)**
- 집계키 유니크 인덱스 위 upsert로 **멱등 재실행** — 2회 실행에도 요약 1,440건·비용 합계 완전 동일
- 가격·환율을 지우지 않고 **SCD-2 버전으로 축적** — 몇 달 전 청구도 그때 가격 그대로 재현

## R — 결과

| 지표 | Before | After |
|---|---|---|
| 조회 속도 p95 | 1,970ms (즉석집계) | **2.6ms** (일배치 사전집계 · 약 750배) |
| 조회 대상 | 원본 3,000,000건 | 요약 **1,440건** |
| 멱등 집계 | — | 2회 실행에도 건수·비용 합계 불변 |
| 엑셀 메모리 (20만 행) | 1,871MB (브라우저) | **8.2MB** (서버 스트리밍) · 50만 행도 22.4MB 평탄 |

- 정산 리포트 수작업 → 자동 생성 대체
- 관리자 직접 조회·내보내기 가능 → 반복 데이터 추출 요청(VOC) 해소

## 예상 질문 방어

- **Q. 왜 사전집계인가?** 매 조회 즉석집계는 원본이 늘수록 선형으로 느려진다. 하루 한 번 요약본만 읽게 해 조회 비용을 상수화했다.
- **Q. Kafka 중복은 어떻게?** 집계키 유니크 인덱스 + upsert. 재실행 자체가 안전한 구조라 장애 복구 시 그냥 다시 돌리면 된다.
