# 파일 업로드 시스템 — UI는 하나, 정책은 주입

## S/T — 왜 공통 컴포넌트인가

같은 다이얼로그인데 상황마다 정책이 달랐다.

| 모드 | 업로드 시점 | 특성 |
|---|---|---|
| 에이전트 등록 | 폼 저장 시점 일괄 (지연) | 폼 수명 주기에 종속 |
| 채팅 첨부 | 세션이 없으면 만들어서 시작 | 세션 의존성 |
| 문서 관리 | 중복 체크 → 사용자 확인 후 즉시 | 중복 정책 |

업로드 함수 1개로는 안 됐다 — **업로드 시점·중복 정책·세션 의존성** 세 가지가 모드마다 달랐고, UI는 한 다이얼로그로 묶여야 했다. UI 일관성과 도메인 분기를 동시에 만족시키는 게 핵심.

## A — 2가지 핵심 설계 결정

**① 공통 컴포넌트 + 외부 정책 주입**
다이얼로그·탭·푸터·상태 표시는 하나의 컴포넌트가 담당. 모드별로 달라지는 부분(엔드포인트·중복 처리·업로드 시점)은 **부모가 모드와 콜백으로 주입**. 타입·유틸·매니저·UI·서비스 5계층으로 책임 분리.

**② 모드별 업로드 시점 분기 (즉시 / 지연 / 확인)**
같은 '업로드' 버튼이지만 누르는 순간 다른 일이 일어난다. → 업로드 시점은 **데이터 수명 주기 + 사용자 인지 부담의 함수**.

### End-to-End 흐름

```
1. 파일 선택/드롭 (확장자·크기·MIME 보정 검증)
2. 모드 분기 (즉시 / 지연 / 확인)
3. 중복 체크 API — 4분기 (HTTP 409 정상 응답)
4. 순차 업로드 루프 — Queued → Uploading → Completed/Skipped/Error
5. 결과 집계 — 전체 성공 / 부분 실패 / 전체 실패
6. 부분 실패 처리 — 모달 유지 → Error 항목 확인 후 재시도
```

## 대표 트러블슈팅

**① HTTP 409 → 비즈니스 정상 응답으로 재해석**
중복 체크는 에러가 아니라 **사실 조회의 결과**다. axios 인터셉터가 거부한 409를 catch에서 다시 꺼내 정상 흐름으로 복귀. 중복 여부 × 덮어쓰기 가능 × 이름 충돌 조합을 4분기로 처리. → HTTP 의미와 비즈니스 의미의 분리.

**② 업로드 중 다이얼로그 닫기**
진행 중 요청을 AbortController로 취소하고 업로드 큐·상태를 정리. 닫힌 뒤 늦게 도착한 콜백이 사라진 화면을 건드리지 않도록 가드. → **중도 이탈을 정상 흐름으로 처리**.

## R — 결과

3 컨텍스트 → 1 컴포넌트 · 4분기 중복 처리 · 6단계 상태 머신으로 부분 실패 회복
