# 서버사이드 PDF 인보이스 — 미리보기와 PDF가 어긋나면 안 되는 정산 문서

## S — 상황

- **정산 신뢰**: 계열사별 토큰 사용량 정산 인보이스 — 미리보기와 실제 PDF가 한 글자도 어긋나면 안 된다
- **한글 깨짐** 허용 불가, **표가 페이지 경계에서 잘리는 것** 허용 불가
- 결국 '내용이 같다'를 사람 눈이 아니라 **구조로 증명**해야 하는 문제

## T — 과제

1. 미리보기와 PDF의 일치를 구조로 보장
2. 한글 폰트·로고·페이지 분할 등 인쇄 품질을 **폐쇄망 컨테이너(musl) 환경**에서 확보
3. '무겁다'를 측정으로 분해해 다음 개선 지점을 숫자로 확정

## A — 행동

### ① 기술 선택 — 후보를 전부 따져보고 Chromium

| 제외 후보 | 이유 |
|---|---|
| jsPDF | 좌표로 직접 그리기 · 한글 폰트 내장 부담 |
| html2canvas | 결과가 이미지라 텍스트 선택 불가 |
| @react-pdf | 화면용·PDF용 두 벌 유지 → 언젠가 갈라짐 |
| wkhtmltopdf | 낡은 엔진, flex/grid 깨짐 |
| 외부 변환 API | 정산 금액 외부 반출 불가 |

**선택**: puppeteer + 헤드리스 Chromium — '보는 화면 그대로'라면 사용자가 보는 그 엔진으로 찍는 게 가장 확실하다.

### ② 구조 뒤집기 — 서버 자기 재진입 제거

처음엔 서버가 PDF를 만들 때 자기 미리보기로 다시 들어가 백엔드를 재호출하고 토큰까지 주입했다. 이걸 뒤집어 **클라이언트가 이미 그린 HTML을 서버로 보내고, 서버는 캡처만** 하게 했다. 재진입·백엔드 재호출·토큰 주입이 전부 사라지고, 보던 화면과 PDF가 같은 DOM이라 **내용·데이터 일치가 그냥 보장**된다.

### ③ 디테일 방어

- **XSS**: JS 실행을 끈 채 캡처해 원천 차단
- **이미지**: 상대경로(로고·아이콘)는 base64 인라인 + 캐시 (8.9ms→1.2ms) · 파일명 영숫자만 허용해 경로 조작 차단
- **타이밍**: 애니메이션 0초 + 한글 폰트 로딩 확인 후 캡처
- **인쇄**: A4 여백 · 행 분할 금지 · 머리글 반복 인쇄 CSS
- **환경**: musl 호환은 시스템 Chromium + 한글 폰트 설치 (→ Dockerfile 고도화와 연결)
- **라우팅**: /api는 인그레스가 백엔드로 보내므로 FE 내부 라우트는 /internal 밑에

### ④ 병목 분해 측정 (멀티페이지 인보이스 · 12회 반복)

| 단계 | 1건당 (p50) | 비중 |
|---|---|---|
| 브라우저 기동 | 4.4초 | **60%** |
| HTML 주입 + 폰트 | 0.3초 | 4% |
| PDF 캡처 | 2.4초 | 32% |
| **전체** | **7.4초** | — |

## R — 결과

- **일치성**: 같은 DOM 캡처로 미리보기 = PDF 내용 일치 · 한글·로고·페이지 분할 정상 (컨테이너 실기동 확인)
- **단순화·보안**: 자기참조·토큰 주입 제거, JS 실행 차단으로 공격면 축소
- **다음 수 확정**: 기동 60%가 지배적 — 브라우저 풀 재사용 시 7.4→2.5초 (66%↓, 측정 확인 · 적용 전) 또는 Gotenberg 전용 서비스 분리

> 데이터를 다시 끌어와 새로 그리는 대신, **이미 그려진 걸 그대로 찍자**고 방향을 뒤집은 게 제일 컸다 — 일치성·보안·단순함이 거기서 같이 따라왔다.
