# RBAC 권한 경계 — 무엇에 묻는지는 FE, 어떻게 강제하는지는 BE

> 리소스 10 × 액션 12(부분집합) · 역할 6 · `shares.shared_with[]`가 SSOT.
> **FE 직접 구현** — can() 추상화 · 3-상태 렌더링 / **BE 설계 이해·연동** — permission.go(멀티소스 SSOT·cascade), ownership-first 소비가 FE·BE 접점.

## 문제 — 기준이 어긋난 두 곳

**서버 · 실시간 연결**: 대화 응답을 멈추거나 세션을 바꾸는 통로가 **로그인 여부만 확인**하고 그 세션이 그 사람 것인지는 안 봤다 → 권한 없는 사람이 남의 대화 응답을 중간에 멈추거나 가로챌 수 있었다.

**화면 · 사이드바**: 에이전트 정보 사이드바 노출을 '대화를 수정할 수 있나(session.update)'로 판단했다. 보여주는 건 에이전트인데 묻는 건 대화 — 기준이 어긋나 권한 없는 정보가 새거나, 봐도 되는데 가려졌다.

**진단**: 두 곳 다 "어느 것에 권한을 걸지"가 실제 보여주는 것과 어긋났다. 그리고 판정 순서가 두 곳에서 같아야 했다 — **주인이면 즉시 통과, 아니면 권한 확인**.

## 행동 — 설계의 무게중심

1. **실시간 통로에 권한 확인 추가 (서버)** — 로그인만 보던 진입점 2곳에 '세션 주인인가 → 아니면 다룰 권한이 있나'를 붙여 우회 경로를 닫았다.
2. **화면 기준을 보여주는 것과 일치** — `session.update` → `agent.read`. 대화는 가졌어도 에이전트 읽기 권한이 없으면 사이드바 미노출.
3. **권한 선(先)체크 → 통과 시에만 데이터 조회** — 권한 없는 데이터는 요청 자체를 안 한다.
4. **3-상태 모델** — `null(조회 전) / false(없음) / true(있음)`. `canReadAgent === true`일 때만 열림 → 확인 중엔 닫혀 있어 깜빡임이 원천부터 없다.
5. **경합(race) 방지** — 에이전트를 연달아 바꾸면 이전 요청의 뒤늦은 응답을 폐기해 최신 화면을 덮지 않게 했다.

```ts
// 역할이 아니라 권한을 묻는다 — 미확정 역할 스펙과 FE 격리
role === 'admin'          // ✗ 역할 하드코딩
can('agent.delete')       // ✓ 역할이 늘어도 FE 무변경
```

## BE — 최종 차단은 서버가 한다 (설계 이해·연동)

화면이 잘 가려도 요청을 직접 쏘면 그만이라, 최종 차단은 서버. 서버도 화면과 같은 순서로 판단한다 — **주인이면 바로 통과, 아니면 공유 목록 확인** (주인일 땐 공유 목록을 아예 안 본다).

- **기준 통일**: 권한은 '공유 목록' 하나로만 판단 (SSOT)
- **조직 격리**: 다른 회사 간 공유는 저장 전 차단
- **정합성**: 권한 변경은 전부 반영 또는 전부 취소 (트랜잭션)
- **연쇄 취소**: 위에서 권한을 거두면 재공유 사슬을 따라 cascade 취소

## 결과

- 권한 없는 사람이 남의 대화 스트림을 제어하던 **경로 2개 제거**
- 읽기 권한자에게만 사이드바 노출 — **정보 누출 차단**, 권한 없음은 팝업 없이 미표시
- 확인 전 닫힘 보장으로 **초기 플리커 소멸**
- 역할이 6개에서 더 늘어도 **화면은 그대로** (서버가 준 '허용된 액션 목록'만 소비)

## 한계와 다시 한다면

| 현재의 한계 | 다시 한다면 |
|---|---|
| 권한 변경 시 새로고침 필요 | 기존 WebSocket으로 즉시 반영 |
| '왜 안 보이지?' 디버깅 어려움 | is_owner / actions / 최종 판정 dev 패널 |
| 같은 권한 매번 재조회 | 호출 증가 시 resource+id 키 캐시 |
| 팀 단위 등 역할 정의 전 | 권한 단위 질의 구조라 그대로 흡수 |

> 세 곳(실시간 연결·화면·데이터)은 같은 권한을 보지 않는다. 같은 건 대상이 아니라 **'주인이면 먼저 통과, 아니면 확인'이라는 순서**다. 권한을 강제하는 건 서버지만, 어느 것에 기준을 걸지는 화면을 만드는 프론트의 판단이었다.
